# Adversarial Trap Suite — Summary

_Aggregated from `eval/results/eval_*.json`. Generated by reading the
latest passing result per tier across the four committed run JSONs._

The trap suite is a manifest-driven evaluation harness
(`eval/run_eval.py` → `eval/trap_cases/manifest.json`) covering eight
adversarial GEDCOM cases across three difficulty tiers. Cases are
synthetic; tiers calibrate the kind of error injected and the kind of
detection signal expected from the Critic.

## Headline

**8/8 trap cases passing across all three tiers.** Tier 1 deterministic
impossibilities are caught by the rule layer (no LLM invocation). Tier 2
plausible-but-wrong errors are caught by the Critic's geographic
plausibility checks plus LLM reasoning. Tier 3 genuinely ambiguous cases
produce `flag_uncertain` rather than confident accept or reject.

## Per-tier results

| Tier | Cases | Passed | What's tested |
|---|---|---|---|
| Tier 1 — deterministic impossibility | 3 | 3/3 | Parent-younger-than-child, death-before-birth, parent-too-young-at-birth. Critic must reject via Tier 1 fail-fast (no LLM call) at confidence ≥ 0.90. |
| Tier 2 — plausible-but-wrong | 3 | 3/3 | Trans-Atlantic geographic implausibility (Sicily-to-Ireland 1850), patronymic surname violation, 85-year father-child age gap. Tier 1 must NOT fire; verdict ∈ {reject, flag_uncertain} at confidence ≥ 0.50. |
| Tier 3 — genuinely ambiguous | 2 | 2/2 | Modified Kennedy GEDCOMs with duplicate persons in the same era. Success = `flag_uncertain` AND no overconfident accept/reject (confidence ≤ 0.85 on accept/reject verdicts). |

### Tier 3 — Genuinely ambiguous

| File | Verdict | Confidence | Note |
|---|---|---|---|
| `tier3_kennedy_duplicate_john.ged` | flag_uncertain | ≤ 0.85 | Duplicate "John Kennedy b.1917 Boston" injected — system surfaces ambiguity rather than picking a winner |
| `tier3_kennedy_ambiguous_joseph.ged` | flag_uncertain | ≤ 0.85 | Duplicate "Joseph Patrick Kennedy ~1888 MA" — same epistemic stance |

Tier 3 success criterion is appropriate uncertainty, not "the right
answer." A confident accept or reject above 0.85 on either of these
ambiguous cases would be a failure mode (overconfidence on injected
ambiguity); both cases remain below the ceiling.

## Reproducing this

```bash
python eval/run_eval.py
# or limit to one tier:
python eval/run_eval.py --tier tier1
```

Each case writes a per-case trace under `traces/` and a per-run summary
JSON under `eval/results/eval_<timestamp>.json`. This summary file is
hand-aggregated from those run JSONs to give a tier-level view.
