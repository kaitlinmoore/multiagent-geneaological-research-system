# Failure Log

Structured table of bugs encountered during Phase 3 development that produced
wrong outputs in ways that survived the unit-test layer and required deeper
investigation. Fields follow the rubric template (Appendix: lightweight
templates, page 16 of the assignment instructions).

For deep narrative analysis of each case — symptom → detection → root cause →
fix → architectural lesson — see `docs/failure_cases.md`.

## Failure cases

| failure_id | date | version_tested | what_triggered_the_problem | what_happened | severity | fix_attempted | current_status |

| **F-01** | 2026-04-25 | Phase 3 dev (DNA Analyst as parallel branch off Scout) | Re-running the same JFK query three times back-to-back during a sanity check before a presentation. | Two runs produced 13 KB reports with Critic verdicts; one produced an 8 KB report with empty `critiques`, `revision_count: 0`, and a final report that was the Hypothesizer's first-pass output dressed up as final. The Critic node had not been entered at all in the short run. Root cause: LangGraph cannot enforce deterministic join semantics when one of the incoming edges to a join node is a conditional edge. The Final Report Writer had two incoming edges (unconditional from DNA Analyst, conditional from Critic's "finalize" path); whenever the ~200 ms DNA branch beat the 30–90 s Critic loop, the Writer fired on stale state. | High — silent verdict suppression that looked superficially complete. | Serialized the DNA Analyst between Scout and Profile Synthesizer instead of running it as a parallel branch. The Hypothesizer reads `state["dna_analysis"]` directly and the Critic uses the `dna_relevant` field. ~200 ms cost, dwarfed by adjacent LLM calls, invisible in practice. | **Fixed.** Verified by re-running the JFK query 10× without recurrence; documented as architectural lesson on conditional-edge join semantics. |
| **F-02** | 2026-04-27 | Phase 3 dev (subtree audit, before surname gate) | Subtree audit run against the Moore family tree (8,759 persons). | A "Joan Knorr" was flagged as a candidate mother for an ancestor whose actual mother was known and present in the tree. The flag carried high enough confidence to appear in the report body rather than the appendix. Root cause: `name_match_score()` weighted given-name and surname tokens equally and combined them with max-of-component across Jaro-Winkler, Levenshtein, Soundex, Metaphone. In a tree of 8,759 persons, the population of "Joan with the right rough birth year" is non-empty by chance; a given-name match plus rough date plausibility was enough to push an unrelated candidate above threshold even with a surname mismatch. | High — wrong-person attribution surfaced as a confident finding on a real family tree. Did not regress on smaller trees (Kennedy/Queen/Habsburg) used during Phase 2 validation. | Added a surname gate at a 0.60 floor in `tools/fuzzy_match.py`: the surname token's independent score must reach 0.60 before composite scoring is even attempted. Threshold chosen empirically — high enough to reject unrelated surnames including phoneme-sharing ones (Knorr vs. Moore), low enough to accept common spelling variants and OCR-corrupted forms (Smyth↔Smith, Kennedy↔Kennedi). | **Fixed.** Re-ran Moore tree audit (no recurrence) and Kennedy/Queen/Habsburg audits (no regression). Architectural lesson: precision at scale requires hard domain-prior gates, not blended composite scores. |
| **F-03** | 2026-04-29 | Phase 3 dev (post DNA-aware Critic integration) | Inspecting the JFK + synthetic DNA demo trace for the Phase 3 report. | Critic structured output contained `verdict: accept` and `confidence_in_critique: 0.95` simultaneously with `dna_relevant.cm_consistency_verdict: "contradicts"` — for a hypothesis with 3487 cM shared, a value squarely in the Parent/Child range of the Shared cM Project. Critic's free-text justification was internally consistent and reasonable; only the structured field was wrong. Root cause: the Hypothesizer emits `proposed_relationship` as natural-language strings (`"father of"`, `"mother of"`); `tools.shared_cm_lookup.is_consistent()` canonicalizes against an alias table that contained `"father"`/`"mother"`/`"parent"` etc. but not the prepositional forms. Lookup returned `consistent=False` with deviation text `"unknown relationship type: 'father of'"`. The Critic's calling code interpreted `consistent==False` as "DNA contradicts" without checking the deviation field. Inter-agent vocabulary drift; structured field disagreed with Critic's own LLM justification but no code checked them against each other. | High — silent inversion of the central agentic claim's output on a flagship demo trace. Bug had been shipping for at least a week before it was noticed because LLM-text justifications looked plausible end-to-end. | Added 30+ relationship-string aliases to `tools/shared_cm_lookup.py::_CANONICAL_ALIASES` covering prepositional forms (`"father of"`, `"mother of"`, `"son of"`, `"daughter of"`, `"parent of"`, `"child of"`), gendered grand- forms, `"aunt of"`/`"uncle of"`, and `"sibling of"`. Alias table now covers every relationship label the Hypothesizer is prompted to emit. Runtime contract assert at the boundary flagged as future work — not yet added. | **Fixed (alias-table layer).** Re-verified on the JFK trace: cm_consistency_verdict now correctly reads `"consistent"`. **Open follow-up:** add typed-enum or runtime-contract layer at the agent boundary to prevent the next instance of vocabulary drift. |
| **F-04** | 2026-04-22 | Phase 3 dev (first Habsburg run) | First attempt to load `data/Habsburg.ged` (Latin-1 encoded, German umlauts: ö, ü, ß). | `python-gedcom-2`'s `Parser.parse_file()` hardcodes `line.decode('utf-8-sig')` and provides no encoding override. The very first header line containing `Österreich` raised `UnicodeDecodeError` on byte 0xF6. Affected every entry-point that parsed the Habsburg tree. | Medium — blocked Habsburg-tree experiments until worked around. Did not silently corrupt data; failed loudly. | Read the file ourselves with `encoding='latin-1', errors='replace'` and call `parse_gedcom_text()` on the resulting string. Workaround applied at every call site rather than upstream patch. Library limitation documented in `tools/gedcom_parser.py`. | **Worked around.** No upstream PR submitted; the workaround is one line at every call site. Acceptable given the project's use of `python-gedcom-2`'s parser is otherwise stable. |
| **F-05** | 2026-04-26 | Phase 3 dev (gap detection initial integration) | Running gap detection on the Moore tree; 1,875 candidate gaps reported in summary table. | Pipeline ran cleanly with no errors but produced zero hypotheses on every gap query. Scout's gap-mode wrote candidate parent records to `state["gap_candidates"]` while the Hypothesizer read from `state["retrieved_records"]`. The graph traversed end-to-end without raising; downstream agents simply received an empty list and returned empty output. | Medium — gap-mode entry path was non-functional. Detected on first integration run; no public traces shipped with the bug. | Normalized the Scout's gap-mode output to write to the standard `retrieved_records` state key (the same key it uses in query mode), so the Hypothesizer reads from a single source of truth regardless of mode. | **Fixed.** Verified on Moore (private) and Kennedy / Habsburg / Queen gap-mode runs. Architectural lesson: state-key contracts between agents need end-to-end integration tests, not just per-agent unit tests. |

## Severity legend

- **Critical**: Wrong outputs at the system's central agentic claim, silent (no error raised). None in this log — F-03 was caught before final submission.
- **High**: Wrong outputs on real data with no error raised; confidence-bearing findings inverted or suppressed.
- **Medium**: Loud failure or non-functional path; obvious during integration.
- **Low**: Cosmetic or non-blocking. None in this log.
